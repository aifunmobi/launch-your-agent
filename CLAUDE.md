# launch-your-agent — "build your local agent" skill

## What this project is
A Claude Code **skill** (`launch-your-agent`) that helps a **technical founder** build whatever they want as a **local Claude Code agent** — an internal worker, a piece of their product, a customer-facing agent; recurring or one-off: interview → agent folder → run → eval → iterate → schedule (when it should run on a clock). The agent runs **on the founder's own machine, inside the Claude Code they're already signed in to** — **no `ANTHROPIC_API_KEY`, no API access, nothing billed per run.** The only prerequisite is the thing they already have to run the skill: Claude Code, installed and signed in. Everything is materialized as Claude Code primitives inside a portable `my-agent/` folder (subagent + grader subagent + slash command + `settings.json`) — drop it anywhere `claude` runs and `/<name>` works. General-purpose, not a workshop — no clock/"hour" framing, no "recurring task" requirement; the spine is an iterative lens (scope a v0, run and grade it, layer upgrades as v1/v2 via NEXT-DIRECTIONS).

## Key decisions
- Audience: technical founder, self-directed, inside Claude Code. Show the machinery (subagent files, `settings.json`, `claude -p`), don't hide it.
- **Build what they need, scoped into versions** (not "defaults until earned"): the starting point is **one subagent** (`.claude/agents/<name>.md` — frontmatter + system-prompt body), **`model: inherit`** by default (inherits the model the founder is already signed into Claude Code with — pin `opus`/`sonnet` only when the use case demands it), the **full Claude Code toolset** (`Bash, Read, Write, Edit, Glob, Grep, WebFetch, WebSearch`), local **`settings.json` permissions** gating which Bash/WebFetch run without asking, an **outcome rubric graded by a local grader subagent** (`.claude/agents/<name>-grader.md`, `max_revisions: 3`), and **drafts-only** — but any primitive (connector, memory, custom skill, schedule) goes in v0 if the core job needs it and it's wireable now; everything else is laid out as numbered v1/v2 increments in NEXT-DIRECTIONS.
- **Menu front door (Phase 0)**: the session opens with a thin, warm AskUserQuestion menu — 🚀 build new · 📋 start from a template · 🔧 run/check an existing `my-agent/` · 📖 what can this do — purely a router. If the founder already stated a clear intent on invocation, skip the menu. "Build new" hands straight to Phase 1's warm open (no second welcome); "template" seeds the build sheet from an examples-bank archetype then runs an abbreviated Phase 1. The menu must stay tiny (one line + four choices) so it doesn't violate "open light".
- **Open light, boundaries in context**: the opening is a couple of warm sentences + 2–3 archetype examples + the open question — no version vocabulary, no boundaries block, no process talk. Caveats/guardrails are raised briefly at the moment they matter. After the founder's first answer, **let them explain** (one open follow-up) before any suggestions, reshaping, or AskUserQuestion menus. Then one cluster at a time, AskUserQuestion for enumerable choices, max one open-ended question per turn.
- **Don't undersell**: keep hard limits (sub-second latency, live phone calls) separate from defaults — building a UI on top of the agent is fine (that's the generated-interface extension, not a limitation); gated write-actions are possible even in v0 if the founder wants them; drafts-first is the recommendation, not a "never".
- **No key handover dance — no Anthropic key, ever**: the agent runs on the founder's existing Claude Code login, so there is **no `ANTHROPIC_API_KEY` to create, fetch, or hand over** and **no waiting on the founder** — they're already signed in. The only secret-hygiene line that remains: if (and only if) the design wires a third-party **connector**, that one optional token goes in `my-agent/.env` (chmod 600, gitignored) and **never into chat**. Don't pre-stage a key table; name the one optional connector secret only if the founder wants it wired now.
- **Connectors are named, not hidden — and mockable**: when delivery is asked for (email/Slack/etc.), state the guardrail (drafts-only + confirm-each-action gate) and the connector route — a **local MCP server** in `settings.json` plus a token in `.env`. Slack and email both take real setup (app/OAuth/webhook + token), so the **default is mock-in-v0, wire-for-real in v1**; if the founder wants to wire it now, do it — pull the latest connector docs first. Don't underbuild: if the core job needs a connector (credential on hand) or a memory store, it goes in v0, not deferred reflexively. Mocking = outbox of schema-true payloads or a stubbed step — `references/mock-connectors.md` has the patterns + typical endpoint schemas — so v1 is just the swap to the real local MCP server + `.env`.
- **Voice**: warm welcome, no clinical process talk, no time estimates or unverified durations; compact/dense messages with tables; consistent emoji shorthand for primitives (🤖 📦 🎯 ▶️ 🗓️ 🔌 🔐 🧠 🧪 🧭) used in the brief, at ✅ checkpoints, and in the recap; **primitives keep their real local names** (Agent = subagent, Environment = machine + `settings.json`, Outcome = rubric + local grader, Run, Schedule = cron/launchd, Connector = local MCP, Secrets = local `.env`, Memory = local folder, Evals — overview cards renamed to match), each with a plain gloss on first use; our own process shorthand stays plain ("the plan / your agent's folder", not "build kit"); deferrals always tagged not-possible vs needs-connector-secret vs out-of-scope; next steps said once at the wrap-up, plus 1–2 extensions tailored to their use case; pre-build read-back is a brief (primitives table + v1/v2 section + eval table) approved via AskUserQuestion. Checkpoint lines use ✅ + emoji + the local artifact, e.g. `✅ 🤖 agent → .claude/agents/competitor-digest.md`.
- **No spend step, no cost notes** — the agent runs on the founder's existing login; nothing is billed per run. The only quiet iteration bound is `max_revisions: 3` on the grade→fix loop. Don't mention costs anywhere unless asked.
- **Next directions** (not "roadmap"/"guardrails"): whenever something doesn't belong in v0, Claude writes it to NEXT-DIRECTIONS.md *in the moment* with what/why-deferred/how, slotted into a numbered version (v1, v2, …) so it reads as planned releases — "not yet" always comes with "here's exactly how, in v1". Mechanisms are written in local terms (add a Slack MCP server + token in `.env`; tighten `settings.json` to allow `WebFetch` only on the competitor domains; re-run `evals/run-evals.sh` before promoting).
- **Celebratory close, owned by `/wrap-up`**: launch-your-agent's Phase 4 finalizes NEXT-DIRECTIONS then invokes `/wrap-up`. The close is a moment, not a checklist read-out: congratulate them on shipping a local agent → overview page reopened → "here's what you built" (primitives recap table: primitive · what it is · what we set · local artifact path · which overview card shows it · unused → NEXT-DIRECTIONS, plus run log) → "here's what's next" (1–2 tailored extensions). Hygiene sweep runs quietly — fix silently, surface only what the founder must act on. Dashboard/recap always show the full model slug when one is pinned (e.g. `claude-opus-4-8`), or `inherit` when the agent rides the session model.
- **Evals**: founder's real past cases = golden set; case 1 is the build input, rest held back; the rubric (via the grader subagent) is the per-run grader; no past cases ⇒ today's verified output is saved as `evals/case-01/expected.md` at close; re-run `evals/run-evals.sh` locally before promoting a new agent version to the schedule.
- **Overview page is describe-only but styled as a live schema** (warm-light schematic: trigger → worker-in-environment → output & grading nodes, run-log lane, version rail — not doc cards; kept skimmable: no never-dos or audience prose on the page), is **regenerated by Claude Code as a workflow step**, and is the **closing interface**: reopened in the browser at the end with run log, verdicts, and schedule. Observability is the local artifacts — `runs/`, `outputs/`, and `agent-overview.html` — not a console. A **generated interface** (graphical output, results viewer, or a way to interact with the agent) is a standard *tailored extension* offer — never a v0 default.
- **Scheduling is local cron / launchd**: when the use case is actually recurring, Phase 4's finale writes a **launchd plist** (macOS `~/Library/LaunchAgents/<label>.plist`, `StartCalendarInterval`) that runs the agent headless via `claude -p`, plus a `crontab` line as the portable option, and offers Claude Code's own `/schedule` as the native route. Trigger it once manually so the founder sees it fire; the run lands in `runs/`. Don't schedule one-off or event-driven agents just to have a finale. Kickoff text must use **relative dates** ("today", "the last 7 days"), never literal ones, since a scheduled job replays it verbatim.
- **Evals run locally**: `evals/` holds case folders (input + expected) and `run-evals.sh`, which loops over each case input, runs the agent, grades with the grader subagent, and collects verdicts into `evals/results-v<N>.md`. No server-side eval run.
- **Sourcing rule** for examples: every interview-menu archetype maps to a real Claude Code primitive (subagent, slash command, settings, headless, MCP, scheduling) — that's its grounding; our own unvalidated ideas live in examples-bank §5 (backlog).

## Testing the skill
- Trigger: `/launch-your-agent` (project or installed copy); `/wrap-up` for the close-out.
- **Run tests inside `test-runs/run-NN/`** (gitignored scratch space, safe to delete) so the generated `./my-agent/` folder doesn't pollute the repo root.
- During a test, log every friction point / missing instruction; fix SKILL.md + references in the **repo** copy, then re-sync to `~/.claude/skills/` so both the repo and installed copies stay in sync (repo is the source of truth).
- Test-run logs and outstanding follow-ups live in `CLAUDE.local.md` (untracked), not here.
